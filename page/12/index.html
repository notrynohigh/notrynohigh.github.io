基于串口的xmodem传输文件协议

在单片机开发中，传输文件基于网络的方式有很多，也可以外接SD卡或者U盘，那么对于只有串口的项目中，希望传输一个文件，比如是字库文件，字库在开发界面时用到的比较多，那么讲字库通过串口传入单片机中然后存入外接的spiflash中也是可行方案。那么推荐一个比较简单的文件传输协议xmodem，这个协议分为xmodem 1k 和 xmodem 128 。
secureCRT软件中有一项菜单是传输，那么子菜单中有xmodem发送，这个是基于xmodem128.那么就讲这个协议了。
先介绍几个命令：
SOH 0x01
EOT 0x04
ACK 0x06
NAK 0x15
CAN 0x18
发送方发送的数据包格式为：
----------------------------------------------------------
| SOH (1B)| 包序号（1B）|包序号补码（1B）|128B数据|校验（1B）|
----------------------------------------------------------
交互流程：
----------------------------------------------------------
接收方 发送方
-> NAK ->
<- |SOH|0X01|0XFE|128bytes|sum| <-
-> ACK ->
<- |SOH|0X02|0XFD|128bytes|sum| <-
-> NAK ->
<- |SOH|0X02|0XFD|128bytes|sum| <-
…
<- EOT <-
-> ACK ->
连接后发送方发送NAK开始传输，在传输过程中每收到一个包回复ACK，如果回复NAK则是重新发送上一包。当最后一包发送成功后，发送方发送EOT代表没有数据发送，则回复ACK结束传输。

Doxygen can help you in three ways: It can generate an on-line documentation browser (in HTML) and/or an off-line reference manual (in LaTeX) from a set of documented source files. There is also support for generating output in RTF (MS-Word), PostScript, hyperlinked PDF, compressed HTML, DocBook and Unix man pages. The documentation is extracted directly from the sources, which makes it much easier to keep the documentation consistent with the source code. cJSON单片机中使用一

利用单片机做项目时很多时候都会用到与上位机通信，运用到通信就会提出如何制定协议。常用的有如下格式： 【头码】【长度】【数据A】【数据B】【数据C】【CRC32】 当单片机以如上格式与上位机软件通信传输数据也是没有任何问题，只需要按照其格式进行解析即可： 1、检测头码 2、获取长度 3、获取所有数据并计算CRC是否正确 4、如果数据接收无误依次获取数据A 数据B 数据C 一般情况下如此设计不会有什么问题，那么为什么还要提一下cJSON呢，以如上的格式进行通信，代码完成后，如果在后续的发展中，产品需要升级迭代，将数据结构变换成： 【头码】【长度】【数据A】【数据B】【数据N】【数据C】【CRC32】 那么在协议解析部分又需要重新编写代码。使用cJSON可以保证在添加数据项后可以不用更改协议解析部分。 例如： ｛ “data_1”: 32, “data_2”: “hello” ｝ 在最初的协议中只有以上两项，上位机传下来的数据通过cJSON解析后，从解析的结果中获取”data_1”和”data_2”两项的值即可。如果日后由于对新产品的需要，上位机传下的数据变更为： ｛ “data_1”: 32, “data_2”: “hello”, “data_3”: “world” ｝ 那么这样的改动对于旧产品来说是没有影响的，就产品收到上位机的数据后同样获取data_1和data_2即可。 如果上面表述有什么问题请在评论区中指出，谢谢！ 下一篇中将用具体代码来测试。

输出打印开源代码easylogger

在代码调试阶段，有的芯片虽然支持在线调试，但是最方便容易定位错误的方式还是在必要的位置添加信息输出。在C语言程序中，大多都是使用printf来进行输出信息。 在代码编写中，包含stdio.h后，完成int fputc(int ch, FILE *pf); 我们可以将串口打印字节的函数用来实现fputc，那么就可以在代码中使用printf进行打印。 这样使用似乎满足了大部分的需求，但在某些情况下，我们需要给打印出的消息分几个级别，比如，information debug warning error assert等。对于每种不同级别的信息我们输出的提示消息不同以便查找错误，同时在代码调试调试结束后提交测试，这个时候可能只需要保留error以上级别的打印消息。那么打印数据就需要可以分级别配置，同时可以限制当前打印的等级。 除了上一点原因，有时候我们也需要给打印输出添加一个buffer，作为缓冲。这样可以不那么频繁的唤醒硬件驱动。 在github上有一个觉得比较不错的log开源代码，easylogger。 [repo owner=”notrynohigh” name=”EasyLogger”] 在port目录下面的c文件中有个输出的函数，根据自己的硬件填写完毕就可以。然后如果使用buffer和async则添加elog_buf.c和elog_async.c 。如果不需要则不添加。同时在inc文件里面也将这两个功能禁止。 在接口方面可以直接调用log_a log_e ..等等，具体可以查看.h文件。 主要还有个问题是log_a没有程序阻塞功能，如果需要可以自己进行改造，也许这也是开源软件的好处，可以自己根据需求去简单的修改代码。 每一次log就是单独的一行，每一行以什么字符或者字符串结尾可以在配置文件中配置。这样带来的一个缺点是有些东西需要单个字符输出但是又不想换行，这个就做不到。只能自己再单独调用底层接口。 总体来说还是很好用的，可以试试。

c语言变长参数琢磨

在C语言中多见的可变参数是在printf中，printf（const char *fmt, …）。 …即是可变参数。

在C库中提供了一个头文件 stdarg.h可以用来提取可变参数。例如：

#include "stdio.h" #include "stdarg.h" void test(char *fmt, ...) { va_list ap; char *p; va_start(ap, fmt); int a = va_arg(ap, int); int b = va_arg(ap, int); char *chr = va_arg(ap, char *); printf("%d %d %s", a, b, chr); va_end(ap); }

int main()
{
test(“%d %d %s\n”, 4, 8, “hello world”);
return 0;
}

通过上面这段代码，最终得到的效果是 4 8 hello world 展示这段代码只是为了记录一下va_start va_arg va_end的用法。

有了这段代码后，就对va_start这个函数的具体代码产生了兴趣，不过暂时还没有找到其函数源码。那么可变参数到底是怎么存放的栈中的呢，形参一般都是存在栈里，从上面的代码可以考虑，fmt的地址可以获取到，那么后面紧接着的可变参数是不是和这个fmt存在一起呢。于是有了下面的代码：
#include "stdio.h" void var_args_func(const char * fmt, ...) { char *ap; int *pint; ap = ((char*)(&fmt));// + sizeof(fmt); pint = (int *)ap; for(int i = 0;i < 20;i++) { printf("%x ", pint[i]); } }

int main()
{
var_args_func(“%d %d %d\n”, 4, 8, 2);
printf(“\n\r”);
var_args_func(“ %d %d %d\n”, 5, 9, 3);
return 0;
}

在获取到fmt的地址后打印紧接着的一块区域。运行效果如下：
400914 0 ff00 0 0 5 3ef55418 7fff 3ef55418 7fff 2f2f2f2f 2f2f2f2f **4 0 8 0 2** 0 4008f0 0 400921 0 ff00 0 8 5 3ef55418 7fff 3ef55418 7fff 2f2f2f2f 2f2f2f2f **5 0 9 0 3** 0 3f16e700 7f9a
在紧接着的区域中确实有我们给的变长参数。看来方向是对的，只是与fmt地址的具体关系还没弄清楚，先留个疑问吧，之后有新的发现再进行记录。如果网友知道的可以在评论区告诉我一下，谢谢。

pt代码实验二

本文主要体验pt代码的demo: example-codelock 在这个例子中使用到了一下几个主要的API： #define PT_WAIT_UNTIL(pt, condition) \ do { \ LC_SET((pt)->lc); \ if(!(condition)) { \ return PT_WAITING; \ } \ } while(0) #define PT_SCHEDULE(f) ((f) == PT_WAITING) PT_SCHEDULE效果是，当f在运行时返回true，当f结束运行返回false. PT_WAIT_UNTIL效果是当其中的condition为false时返回，相当于代码阻塞在此处等待condition为true。 看看在demo代码中如何使用这两个宏。 int main(void) { PT_INIT(&input_pt); PT_INIT(&codelock_pt); while(PT_SCHEDULE(codelock_thread(&codelock_pt))) { PT_SCHEDULE(input_thread(&input_pt)); #ifdef _WIN32 Sleep(0); #else usleep(10); #endif } return 0; } main函数中初始化了两个pt结构，分别用在两个pt thread，初步查看main函数可以大致猜想其功能： codelock_thread()在其内部会有一个地方被阻塞，阻塞时等同于while(1)，那么就会执行下面的input_thread函数。那么在input_thread函数中应该会产生某个调节使得codelock_thread继续执行，当codelock_thread执行结束后demo程序结束。 代码贴上来太占地方，大家查看源码可以访问本站另一篇文章，《pt代码实验一》 代码中大致的过程是，input_thread来模拟按键数据，codelock_thread来检测按键的输入与本地保存的键值是否相符。 代码运行后的效果如下，有兴趣的可以对照着运行结果回头查看代码： Waiting 1 second before entering first key. --- Key '1' pressed Correct key '1' found --- Key '2' pressed Incorrect key '2' found --- Key '3' pressed Incorrect key '3' found --- Key '1' pressed Correct key '1' found --- Key '4' pressed Correct key '4' found --- Key '2' pressed Correct key '2' found Code lock timer expired. --- Key '3' pressed Incorrect key '3' found --- Key '1' pressed Correct key '1' found --- Key '4' pressed Correct key '4' found --- Key '2' pressed Correct key '2' found --- Key '3' pressed Correct key '3' found Correct code entered, waiting for 500 ms before unlocking. --- Key '4' pressed Key pressed during final wait, code lock locked again. --- Key '1' pressed Correct key '1' found --- Key '4' pressed Correct key '4' found --- Key '2' pressed Correct key '2' found --- Key '3' pressed Correct key '3' found Correct code entered, waiting for 500 ms before unlocking.
